# 자기 평가서 (self_review.md)

컴퓨터공학과 202620855 김한결

## 1. 가장 잘했다고 생각하는 부분

가장 잘했다고 생각하는 부분은 검증 로직을 `Transaction` 부모 클래스 한 곳에
모은 설계입니다. 수입(`Income`)과 지출(`Expense`)은 모두 날짜와 금액 검증이
필요한데, 같은 코드를 자식 클래스마다 반복하지 않고 부모의
`_validate_amount()`와 `_validate_date()`에 모아 두었습니다. 덕분에 자식
클래스는 `super().__init__()`을 호출한 뒤 자기에게 필요한 속성(출처,
카테고리)만 추가하면 되고, 검증 규칙이 바뀌어도 '한 곳만 고치면 되는
구조'가 되었습니다. 또 `signed_amount()`를 수입은 양수, 지출은 음수로
구현해서, `Ledger`가 거래 종류를 일일이 구분하지 않고 단순한 합산만으로
잔액을 계산하게 만들었고, 이 과정에서 상속과 다형성이 코드 중복을 줄이기
위한 실용적인 도구라는 점을 체감했습니다.

두 번째는 검증 습관입니다. pytest 19개(정상 케이스와 엣지 케이스),
doctest 40개, pycodestyle 경고 0건을 유지했고, 새 가상환경에서
`pip install .` 후 README의 빠른 시작 코드가 그대로 동작하는지까지 확인한
뒤에 제출물을 정리했습니다. 이 과정을 반복하면서, 단순히 '동작하는 코드'와
'신뢰할 수 있는 코드'의 차이는 결국 테스트와 검증의 습관에서 나온다는
점을 깨달았습니다.

## 2. 아쉬운 부분

가장 아쉬운 부분은 저장 기능이 없다는 점입니다. 지금은 거래가 메모리에만
존재해서 프로그램을 끄면 기록이 사라집니다. 가계부라는 주제의 특성상 파일
저장이 있어야 실제로 쓸 수 있는 도구가 되는데, 강의에서 배운 `open()`
기초만으로도 구현할 수 있었음에도 범위 조절을 하느라 넣지 못했습니다.

날짜 검증의 한계도 아쉽습니다. 현재 `_validate_date()`는 'YYYY-MM-DD'
형식과 월(1~12), 일(1~31) 범위만 확인하기 때문에, 2026-02-30처럼 실제로
존재하지 않는 날짜는 걸러내지 못합니다. 형식 검사와 실제 달력 검사는
다르다는 것을 이번에 알게 되었습니다.

## 3. 다음에 개선하고 싶은 점

먼저 CSV 저장·불러오기 기능을 추가하고 싶습니다. `Ledger`에 `save(path)`와
`load(path)` 메서드를 만들어 거래 내역을 파일로 보관하면 실제 가계부로 쓸
수 있게 됩니다. 다음으로 날짜 검증을 달별 일수까지 확인하도록 보강하고,
카테고리별 예산 한도를 정해 두면 초과할 때 경고해 주는 기능도 붙여 보고
싶습니다. 마지막으로 이번에 처음 써 본 GitHub Actions CI를 발전시켜서
테스트 커버리지 측정까지 자동화해 보고 싶습니다.

아직 부족한 점이 많지만, 이번 과제에서 익힌 패키지 구조와 테스트 습관을
발판으로, 아이디어를 직접 결과물로 만들어내는 개발자로 성장하고 싶습니다.

## 4. AI 도구 사용 내역

이 프로젝트에서는 Claude Code(Anthropic)를 사용했습니다. 제가 생각하는
AI 활용의 핵심은 AI가 아니라 사람이 주체가 되어야 한다는 것입니다. AI가
처음부터 끝까지 대신 만들어주면 결과물이 제 실력에 비해 과해지고, 정작
제가 설명하지 못하는 상황이 생기기 때문입니다. 그래서 작업을 시작하기
전에 수강한 강의 8개의 범위를 직접 정리해서, 타입 힌트, dataclass,
데코레이터, pytest.raises, pathlib, 외부 라이브러리처럼 강의에서 다루지
않은 문법을 금지하는 제약 가이드를 먼저 만들어 적용했습니다.

구체적인 활용 내역은 다음과 같습니다 (질문 → 답 → 활용 순).

- **설계와 초안 작성**: 과제 요구사항(클래스 2개 이상, 상속, 비공개
  메서드, 테스트 8개 이상)을 어떻게 만족시킬지 질문했고, `Transaction`
  부모 클래스 + `Income`/`Expense` 자식 클래스 + `Ledger` 장부 클래스
  구조의 코드 초안을 위 제약 조건 아래에서 생성받았습니다. 생성된 코드는
  모듈별로 직접 읽으면서 각 클래스의 책임과 메서드 동작을 확인한 뒤에
  채택했습니다.
- **설계 이유 질문**: 검증 로직을 왜 부모 클래스에 두는지,
  `super().__init__()`이 없으면 어떤 문제가 생기는지, `signed_amount()`를
  자식마다 다르게 구현한 이유가 무엇인지 질문해서 DRY 원칙과 다형성
  관점의 설명을 들었습니다. 이를 바탕으로 코드 이해 자가 점검 질문 목록을
  만들어 스스로 답해 보면서 이해했는지 확인했습니다.
- **검증 실행**: pytest(19개), pycodestyle(경고 0건), doctest(40개), 새
  가상환경에서의 `pip install .`과 README 빠른 시작 코드 실행을 반복
  수행했고, 결과를 results.md에 정리했습니다.
- **직접 수정·결정한 부분**: README 빠른 시작 예시의 월간 리포트 예상
  출력이 실제와 다른 것(지출 50,000원 → 51,500원)을 제가 직접 계산해서
  발견하고 수정했습니다. 작성자 정보는 GitHub에서 직접 입력했고, 가상환경
  재구성(삭제 후 재생성·설치·테스트)도 직접 수행했습니다. 코드 주석과
  커밋 메시지를 한글로 쓰기로 한 것도 제 결정입니다.
- **AI 사용 기록**: 작업할 때마다 ai_log.md에 무엇을 요청했고 무엇을
  확인했는지 기록했고, 이 문서는 그 기록의 요약입니다.

이 과정에서 배운 점은 AI가 생성한 코드와 문서도 그대로 믿으면 안 된다는
것입니다. 실제로 문서의 수치 오류를 직접 발견해서 고쳤고, 모든 코드는
실행과 테스트로 검증한 뒤에만 커밋했습니다. AI에 대체되는 개발자가 아니라
AI를 가장 잘 다루는 개발자가 되겠다는 생각으로, 이번 과제에서도 '사람이
주체가 되는 AI 활용'이라는 원칙을 지키려고 노력했습니다.

### 참고 출처

- 사용한 문법은 모두 강의(DataCamp) 범위 내: 클래스/상속/super(),
  비공개 메서드, try/except, raise, f-string, collections.defaultdict,
  리스트 컴프리헨션, setuptools.setup
- 단, utils.format_amount의 천 단위 콤마 포맷 지정자 `f"{amount:,}"`는
  강의에서 직접 다루지 않아 Python 공식 문서(Format Specification
  Mini-Language)를 참고해 사용함
- 외부 라이브러리 미사용 (pytest, pycodestyle은 개발 도구로만 사용)
